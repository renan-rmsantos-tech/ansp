# ADR-003: Direct-to-Storage File Uploads with Signed URLs

## Status

Accepted

## Date

2026-06-16

## Context

The scholarship form collects multiple document uploads per application (parent RG, marriage certificate, proof of address, student documents, income tax statement, bank statements). Applicant families often access the form on mobile devices with slow connections. The upload strategy must handle multiple files per step without blocking form progression, support files up to ~10MB (scanned documents, photos), and store files securely so only authenticated admins can retrieve them.

## Decision

Use direct client-to-Supabase-Storage uploads via signed upload URLs. The flow:

1. Client requests a signed upload URL from a Next.js API route (or Server Action)
2. Server generates a time-limited signed URL for a specific path in the `documents` bucket
3. Client uploads the file directly to Supabase Storage using the signed URL
4. Client stores the returned file path in form state
5. On form submission, file paths are saved with the application record in Postgres

## Alternatives Considered

### Alternative 1: Server-Side Proxy Upload

- **Description**: Files uploaded to a Next.js API route, which then streams them to Supabase Storage
- **Pros**: Server can validate file content (type, size, malware scan) before storage
- **Cons**: Doubles bandwidth (client → server → storage), adds latency, Vercel serverless functions have 4.5MB body limit on free tier (50MB on Pro), memory pressure for large files
- **Why rejected**: The 4.5MB body limit on Vercel free tier is a hard blocker for document uploads; even on Pro, the double-hop latency hurts mobile users on slow connections

### Alternative 2: Upload All Files on Final Submit

- **Description**: Collect files in browser memory across all steps, upload everything when the user submits at Step 6
- **Pros**: Simpler implementation, files only stored for completed applications
- **Cons**: High risk of failure on slow connections with many files (could be 8-12 files totaling 50MB+), browser memory pressure, a single upload failure loses everything
- **Why rejected**: Unacceptable UX risk for the target audience (families on mobile)

## Consequences

### Positive

- No server-side bandwidth or memory overhead for file uploads
- Files upload as the user progresses through each step, not all at once
- Failed uploads can be retried individually without losing other progress
- Signed URLs are time-limited and path-scoped, preventing unauthorized uploads

### Negative

- Server-side file content validation is deferred (can only validate metadata and extension, not actual file content)
- Orphaned files may accumulate if users start but don't complete the form — requires periodic cleanup

### Risks

- Orphaned files from incomplete submissions consume storage. Mitigation: scheduled cleanup job (cron or Supabase Edge Function) that deletes files not referenced by any submitted application after 24-48 hours
- Signed URL could theoretically be shared. Mitigation: URLs expire in 5 minutes and are scoped to a unique path per upload

## Implementation Notes

- Supabase Storage bucket `documents` with RLS: anonymous users can upload to `pending/{uuid}/*`; only authenticated admin role can read/download
- On form submission, files are moved (or paths updated) from `pending/{uuid}/` to `applications/{application_id}/`
- File type restriction via Supabase Storage policies: `image/*`, `application/pdf` only
- Max file size: 10MB per file (configurable via Storage policy)
