# ADR-002: Next.js App Router + Supabase + shadcn/ui Tech Stack

## Status

Accepted

## Date

2026-06-16

## Context

The ANSP Scholarship Management System is a greenfield web application with two interfaces: a public multi-step form and a private admin panel. It needs a cloud database (Postgres preferred), file storage for document uploads, built-in authentication for committee members, and a component library that can be customized to match the institutional brand spec (navy, gold, warm paper, serif headings). The target audience includes families on mobile devices and 2-5 admin users. Deployment should be straightforward with minimal infrastructure management.

## Decision

Use the following tech stack:

- **Framework:** Next.js 15 with App Router (React Server Components, Server Actions)
- **Database + Auth + Storage:** Supabase (Postgres, Supabase Auth, Supabase Storage) — São Paulo region
- **UI:** shadcn/ui components + Tailwind CSS, customized to match the brand spec
- **Form handling:** Server Actions with `useActionState` for multi-step form progression, Zod for schema validation
- **File uploads:** Direct client-to-Supabase-Storage via signed upload URLs
- **Deployment:** Vercel

## Alternatives Considered

### Alternative 1: Remix + Firebase

- **Description**: Remix for progressive enhancement and strong form handling, Firebase for NoSQL storage and auth
- **Pros**: Excellent form handling primitives, progressive enhancement by default
- **Cons**: Smaller ecosystem, NoSQL less natural for relational scholarship data (families → students → decisions), Firebase query limitations for admin filtering
- **Why rejected**: Relational data model is a better fit for Postgres; Next.js has broader community support and Vercel deployment is turnkey

### Alternative 2: Next.js + Neon Postgres + Vercel Blob

- **Description**: Serverless Postgres (Neon) for data, Vercel Blob for file storage, custom auth
- **Pros**: Tight Vercel integration, serverless scaling
- **Cons**: No built-in auth — would need NextAuth.js or custom implementation. Vercel Blob has no SA region. Two separate services to manage vs. Supabase's integrated platform
- **Why rejected**: Supabase provides auth + storage + database in one platform with a São Paulo region, reducing integration complexity

### Alternative 3: Custom CSS Modules instead of shadcn/ui + Tailwind

- **Description**: Hand-written CSS matching the existing inline styles from HTML prototypes
- **Pros**: Direct 1:1 translation of prototype styles, no framework overhead
- **Cons**: No accessible component primitives (must build from scratch), slower development, higher maintenance burden
- **Why rejected**: shadcn/ui provides Radix-based accessible primitives (dialog, dropdown, tabs) that would be costly to rebuild; Tailwind config maps cleanly to the design tokens

## Consequences

### Positive

- Supabase's integrated platform (DB + Auth + Storage) reduces the number of services to configure and maintain
- São Paulo region for Supabase satisfies LGPD data residency preference
- Row Level Security (RLS) in Supabase Postgres provides defense-in-depth for data access control
- shadcn/ui components are copy-pasted into the project (no runtime dependency), fully customizable
- Server Actions simplify form submission without manual API route wiring
- Vercel deployment provides preview deployments for testing, CDN, and HTTPS by default

### Negative

- Supabase free tier has storage and bandwidth limits that may need upgrading if document uploads are heavy
- Server Actions for multi-step forms require careful state management across steps (client-side state still needed between steps)
- shadcn/ui components will need non-trivial theme customization to match the institutional brand (OKLch colors, serif display font)

### Risks

- Supabase outage would make the entire system unavailable. Mitigation: Supabase has 99.9% uptime SLA on paid plans; MVP can start on free tier and upgrade before enrollment window
- Direct-to-storage uploads bypass server-side file validation. Mitigation: use Supabase Storage policies to restrict file types and sizes; validate metadata on form submission

## Implementation Notes

- Tailwind config will define custom theme tokens matching `docs/brand-spec.md`: `--accent`, `--gold`, `--bg`, `--surface`, etc.
- Supabase client initialized with `@supabase/ssr` for Next.js App Router compatibility
- Admin auth uses Supabase Auth with email/password provider; committee accounts created via Supabase dashboard or seed script
- Public form does not require authentication; anonymous uploads use a Supabase Storage bucket with upload-only RLS policy
